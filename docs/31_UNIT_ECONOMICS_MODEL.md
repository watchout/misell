# 31. ユニットエコノミクス設計

## 目的

Misellを継続運用できる事業にするには、1施設あたりの採算、運用負荷、更新頻度、契約期間、広告運用範囲を管理する必要がある。

この文書はpublic repositoryに残せる採算管理の考え方だけを定義する。実採算、仕入れ、施工、パートナー条件、チャネル別条件はpublic repositoryに置かない。

## データの置き場所

| データ種別 | public repository | private store |
| --- | --- | --- |
| 顧客提示用プラン価格 | `docs/data/pricing/misell_pricing.public.yaml` | 人間承認履歴 |
| 含まれる範囲/含まれない範囲 | `docs/data/pricing/misell_pricing.public.yaml` | 案件別の見積補足 |
| 実採算、仕入れ、施工条件 | 置かない | 必須 |
| パートナー条件、チャネル別条件 | 置かない | 必須 |
| 見積前提、標準外作業 | 参照方針のみ | 案件別詳細 |

## 収益項目

### 初期収益

- 端末
- ディスプレイ
- 金具
- キッティング
- 初期設定
- 施工
- 初期コンテンツ制作

### 月額収益

- 配信管理
- 監視
- 保守
- 素材更新
- レポート
- 広告運用
- AI制作

### 追加収益

- 動画制作
- POPタグ制作
- 広告主LP制作
- 匿名集計を前提にしたAI/センサー拡張
- 電子棚札連携
- 多店舗本部管理

## 採算モデルの管理方針

Lite、Standard、Mediaごとの顧客提示価格はpublic pricing datasetを参照する。採算モデルはprivate storeで、案件条件、運用範囲、施工難易度、更新回数、外部パートナー利用有無を分けて管理する。

public repositoryでは、以下のような採算値を直接書かない。

- 仕入れや施工の実数
- 月次運用の実数
- 商品別の採算値
- チャネル別の条件
- パートナー条件
- 率や取り分

## 見るべき指標

private storeでは、少なくとも以下を追う。

- 施設別MRR
- 契約期間
- 解約理由
- 月次更新回数
- 素材制作工数
- 監視/保守対応件数
- 標準外作業の発生理由
- 広告主数
- 広告掲載継続率
- QR読み取り数
- レポート提出有無

## 改善レバー

- 月次更新回数を標準化する
- AIレポートをログに基づく文章化支援として使う
- POPタグを追加する
- 広告主を獲得する
- 多店舗管理費を設計する
- 初期制作をテンプレート化する
- 施工を標準化する
- 標準外作業を見積前に切り分ける

## 営業上の注意

- 広告収益、来館者数、売上増加、QR数、視認者数は保証しない
- AIは数値を作らず、既存ログに基づく文章化と改善仮説に限定する
- AIカメラや匿名カウントは、法務、プライバシー掲示、設置条件の確認後に個別提案する
- 顧客提示用価格は `docs/data/pricing/misell_pricing.public.yaml` を参照し、資料ごとに手書きで複製しない
