# 06. Business Model & Pricing

## 基本方針

Misellは、機器販売ではなく、施設内メディア運用として販売する。

顧客提示用価格の正本は `docs/data/pricing/misell_pricing.public.yaml` とする。この文書は、価格表そのものではなく、収益レイヤー、販売方針、内部採算データの扱いを整理するための参照メモである。

public repositoryには、顧客へ提示できる価格、含まれる範囲、含まれない範囲、source statusだけを置く。実採算、仕入れ、施工、パートナー条件、チャネル別条件はprivate repository、local database、または承認済みのignored local storeで管理する。

## 収益レイヤー

### 1. 初期導入

- 端末
- ディスプレイ
- 金具
- 配線
- キッティング
- 初期設定
- 設置施工
- 初期テンプレート制作

### 2. 月額運用

- 配信管理
- 素材更新
- 死活監視
- 一次保守
- 軽微なコンテンツ修正
- 月次レポート

### 3. 広告運用

- 地域広告枠設計
- 広告主対応範囲の整理
- クリエイティブ制作
- QR計測
- 掲載レポート
- 改善提案

### 4. AI/データアップセル

- AIコンテンツ生成
- AIレポート
- 配信改善提案
- 匿名集計を前提にした人流/滞留分析
- 多言語案内
- POS/在庫/予約連携
- POPタグ/電子棚札連動

## 料金プランの扱い

Lite、Standard、Mediaの顧客提示用価格は、手書きの表ではなく `docs/data/pricing/misell_pricing.public.yaml` を参照する。

提案資料、LP、申込書、見積テンプレートでは、以下を守る。

- public datasetの `source_status` が `needs_human_approval` の価格は、正式価格ではなく承認待ちとして扱う
- 価格、含まれる範囲、含まれない範囲を資料ごとに複製しない
- AI機能は、制作支援、レポート文生成、改善提案から説明する
- AIカメラ、センサー、匿名カウントは、法務、プライバシー掲示、設置条件の確認後に個別提案する
- 広告収益、来館者数、売上増加、QR数、視認者数は保証しない

## Rasenとの境界

Rasenへ渡せるものは、schema、template、adapter、抽象的な価格データ構造までとする。

Misellの商品固有価格、実採算、仕入れ、施工、パートナー条件、チャネル別条件はRasenへ渡さない。

## 施工価格方針

標準施工範囲と標準外作業を明確に分ける。

- 標準施工範囲はpublic pricing datasetのプラン説明と見積条件に接続する
- 標準外作業は個別見積にする
- 自社施工と外注施工の違いは、顧客提示価格ではなく内部採算管理で扱う
- 施工方法、現場条件、端末構成、台数変更は正式見積前に確認する

## 内部採算管理

内部採算はpublic repositoryでは管理しない。

管理対象の例:

- 商品別の採算
- 施工管理の採算
- 自社施工時の差分
- 月額運用の採算
- 広告運用の採算
- AI/データアップセルの採算
- パートナー条件

これらはprivate data storeに分離し、公開資料では数値、比率、条件を直接書かない。

## 監査

pricing実装PRでは、以下を必ず実行する。

```bash
npm run pricing:audit
```

このauditは、public pricing datasetのschema validation、内部データ混入チェック、古い手書き価格表チェック、sales claimチェックを行う。

## 注意

初期費用だけで売ると施工案件化しやすい。Misellの価値は、施設内の案内、販促、広告枠、月次運用、計測、改善提案を継続的に回せることに置く。
